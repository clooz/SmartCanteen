# 管理端权限目录（细分方案）

**版本**：2026-05-15  
**说明**：本文件描述后台 RBAC **permission key** 的产品与技术约定（建议的「v2 细分」粒度）。实现时须与代码中的权限常量 **[server/src/constants/permissions.js](../server/src/constants/permissions.js)** 对齐并保持单一事实来源。  

**与总方案的关系**：边界、超管、403、审计等见 **[管理端_RBAC_权限方案.md](./管理端_RBAC_权限方案.md)**；本文只谈「每个模块有哪些操作级权限、如何命名、如何与旧 key 兼容」。

---

## 1. 目的与范围

- **约束对象**：管理端 HTTP API（Express 路由 + controller）与 **admin-web** 的菜单、路由守卫、按钮级展示。
- **不替代**：`users.role`（`employee` / `chef` / `admin`）的**准入**含义；后台能力仍以「能进管理端 + `admin_role` 权限集合」组合判定。
- **不在此文档**：具体中间件改写步骤、完整 PR 清单；落地时按模块改路由并回归测试即可。

---

## 2. 命名与粒度惯例

| 原则 | 说明 |
|------|------|
| 格式 | `资源[:子资源]:动作`，片段小写，与现有 `orders:status:update`、 `companies:delete` 一致。 |
| `read` | 列表、详情、只读配置查询；不隐含写入。 |
| `create` / `update` / `delete` | 与常见 REST 语义对齐；合并「关闭/重开」等可视为 `update`（活动状态）。 |
| 高风险 / 工作流 | 审核、批量同步、重置密码、改订餐默认、报表等，**宜单独 key**，避免一个 `write` 包办。 |
| 粒度上限 | 不为每个 UI 按钮各建一个权限；仅当**不同岗位需要不同授权**时再拆。 |
| 仅超管模块 | `rbac:*`、`audit:read` 受众单一，可保持较粗，不必拆成十余个 CRUD key。 |

---

## 3. 与当前实现对照（摘要）

当前 **[permissions.js](../server/src/constants/permissions.js)** 中部分模块使用 `*:write` 合并多项操作，建议逐步拆细：

| 现状（粗） | 拟拆方向 |
|------------|----------|
| `dishes:write` | `dishes:create`、`dishes:update`、`dishes:delete` |
| `menus:write` | `menus:create`、`menus:update`、`menus:delete`，另加 `menus:ordering_defaults:update`（或 `menus:defaults:update`） |
| `users:write` | `users:create`、`users:update`、`users:password:reset` |
| `wish:manage` | `wish:activities:manage`、`wish:items:adopt`（另见下节许愿） |
| `kitchen:*` 与 `orders:status:update` | 需对齐（见第 4.4 节），避免目录与校验不一致 |

## 4. 按模块的建议权限

### 4.1 菜品 `dishes`

| 建议 key | 说明 |
|----------|------|
| `dishes:read` | 查看列表、详情、分类（管理端与 `allowEmployeeOrPermission` 场景以代码为准） |
| `dishes:create` | 新增菜品 |
| `dishes:update` | 编辑菜品（含图片） |
| `dishes:delete` | 删除菜品 |

**路由对应（概念）**：`POST/PUT/DELETE /api/.../dishes` 见 **[server/src/routes/dishes.js](../server/src/routes/dishes.js)**。

---

### 4.2 菜单 `menus`

| 建议 key | 说明 |
|----------|------|
| `menus:read` | 查看菜单与订餐默认 |
| `menus:create` | 新建菜单 |
| `menus:update` | 修改菜单主体内容 |
| `menus:delete` | 删除菜单 |
| `menus:status:update` | 发布/状态流转（`PUT .../status`）— 若产品不需要「仅能改状态、不能改内容」的岗位，可合并进 `menus:update` |
| `menus:ordering_defaults:update` | 修改订餐默认（`PUT /ordering-defaults`） |

**路由对应（概念）**：**[server/src/routes/menus.js](../server/src/routes/menus.js)**。

---

### 4.3 订单与报表 `orders`

| 建议 key | 说明 |
|----------|------|
| `orders:read` | 全部订单列表 |
| `orders:report` | 消费统计报表 |
| `orders:status:update` | 后台/厨房侧更新订单状态（非员工取消规则） |

**说明**：员工取消 `pending` 订单等逻辑在 **[orderController.js](../server/src/controllers/orderController.js)** 的 `updateOrderStatus` 内按 `users.role` 分支；管理端改状态需具备 `orders:status:update`。

---

### 4.4 厨房 `kitchen` 与 `orders` 对齐策略（二选一）

**现状**：权限目录中存在 `kitchen:view`、`kitchen:order:update`，但 **`updateOrderStatus` 对 chef/admin 实际校验的是 `orders:status:update`**，易出现「库里配置了 kitchen 却与后端不一致」的问题。

推荐任选其一并在实现与文档中统一：

- **策略 A（推荐）— 单一事实来源**  
  - 废弃或不再分配 `kitchen:*`。  
  - 厨房页/实时订单：`orders:read`（可见）+ `orders:status:update`（可改状态）。  
  - 前端 **[App.tsx](../admin-web/src/App.tsx)** / **[AppLayout.tsx](../admin-web/src/components/AppLayout.tsx)** 的 `permission` 改为上述 key。

- **策略 B — 保留 `kitchen:*` 作为别名**  
  - 目录仍可展示 `kitchen:view`、`kitchen:order:update`，但在 **[rbacService](../server/src/services/rbacService.js)**（或等价解析层）将二者**展开/映射**为 `orders:read` 与 `orders:status:update`，保证**接口校验与展示语义一致**。

---

### 4.5 许愿 `wish`

管理端写操作建议从单一 `wish:manage` 拆为（**折中粒度**，可平衡配置成本）：

| 建议 key | 说明 |
|----------|------|
| `wish:read` | 后台查看活动与条目（若存在「只读运营」岗位；否则可与下面权限一起授） |
| `wish:activities:manage` | 创建活动、编辑、关闭、重开 |
| `wish:items:adopt` | 采纳愿望条目 |

**备选（更细）**：将 `wish:activities:manage` 再拆为 `wish:activities:create`、`wish:activities:update`，仅在岗位划分确实需要时再采用。

**路由对应（概念）**：**[server/src/routes/wish.js](../server/src/routes/wish.js)**。

---

### 4.6 用户 `users`

| 建议 key | 说明 |
|----------|------|
| `users:read` | 用户列表与详情 |
| `users:create` | 新建用户 |
| `users:update` | 编辑用户信息（非密码） |
| `users:password:reset` | 重置密码接口 |
| `users:sync` | 外部批量同步（高危，保持独立） |

**说明**：当前 **[admin.js](../server/src/routes/admin.js)** 无删除用户路由，**不设** `users:delete`，除非产品补充删除能力。

---

### 4.7 公司 `companies`

| 建议 key | 说明 |
|----------|------|
| `companies:read` | 查看 |
| `companies:create` / `companies:update` | 可选：由现有 `companies:write` 拆成二者，便于「可建不可开」类岗位 |
| `companies:delete` | 删除（已独立，保留） |

若暂不拆 `write`，可继续沿用 `companies:write` 表示 create+update，与 **admin 路由**一致后再迭代。

---

### 4.8 充值 `recharge`

| 建议 key | 说明 |
|----------|------|
| `recharge:read` | 查看申请列表 |
| `recharge:review` | 审核通过/驳回 |

已与「查看 / 审核」分离，**建议保持**。

---

### 4.9 RBAC 与审计 `rbac` / `audit`

| 建议 key | 说明 |
|----------|------|
| `rbac:role:manage` | 岗位与权限树维护 |
| `rbac:assign` | 为用户绑定岗位 |
| `audit:read` | 查看操作日志 |

**说明**：上述能力按总方案通常**仅超级管理员**；无需再拆 CRUD 级 RBAC key。

## 5. 兼容与迁移策略

1. **目录双轨**：在 `PERMISSIONS` 中为新 key 增加元数据；旧 key（如 `dishes:write`）标注 deprecated，保留一至两个版本周期。
2. **解析层展开（推荐）**：在 **`resolveUserRbac`** 生成 `permissionSet` 时，若存在旧 key，自动并入对应新 key（或在校验时同时认可旧 key），减少一次性 SQL 批量更新。
3. **路由中间件**：各 `requirePermission` 改为新 key；过渡期可使用「新 key **或** 旧 key」双许可，稳定后移除旧 key。
4. **种子角色**：`SYSTEM_ADMIN_KEYS`、`CHEF_DEFAULT_KEYS` 等改为新 key 集合；存量库可用迁移脚本或管理端重新保存角色权限。
5. **前端**：`authStore` / `hasPermission` 与后端 key 对齐；权限配置页按**模块分组**展示子项。
